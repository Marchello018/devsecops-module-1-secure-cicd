# Модуль 1. CI/CD Security

## О модуле

Это первый практический модуль по направлению DevSecOps. Его задача не просто показать, как работает GitHub Actions, а научить смотреть на CI/CD как на часть attack surface. В современном проекте pipeline имеет доступ к коду, зависимостям, артефактам, секретам и иногда к production-инфраструктуре. Поэтому компрометация CI/CD часто означает компрометацию всей системы.

Главная идея модуля: pipeline должен быть не просто автоматизированным, а безопасным по умолчанию.

---

## Цель модуля

После прохождения этого модуля я должен уметь:

- объяснить, почему CI/CD является частью attack surface
- перечислить основные угрозы для pipeline
- настроить безопасный GitHub Actions workflow
- разделить trusted и untrusted execution
- применять least privilege в CI/CD
- запускать тесты, security scan и Docker build в одном пайплайне
- объяснить это решение на собеседовании

---

## Почему CI/CD нужно защищать

Многие воспринимают CI/CD как “скрипт сборки”, но на практике это критическая точка доверия. Pipeline получает доступ к:

- исходному коду
- зависимостям
- конфигурации
- build-артефактам
- токенам и секретам
- контейнерным registry
- deployment-механизмам

Если атакующий получает контроль над CI/CD, он может:

- внедрить вредоносный код в проект
- украсть секреты
- подменить артефакт перед деплоем
- выполнить несанкционированный деплой
- использовать runner как точку входа во внутреннюю инфраструктуру

## Основные угрозы CI/CD

### 1. Утечка секретов

Секреты могут утечь:

- через логи
- через `echo`
- через debug output
- через небезопасные third-party actions
- через переменные окружения в untrusted workflow

### 2. Злоумышленный pull request

Особенно опасны PR из fork. Атакующий может прислать код, который выполнится в CI, и если workflow настроен неправильно, то сможет:

- прочитать секреты
- выполнить произвольные команды
- попытаться подменить артефакты

### 3. Third-party actions

Любой `uses:` в GitHub Actions означает запуск чужого кода. Риск:

- action может быть вредоносным
- action может быть скомпрометирован
- владелец action может изменить содержимое тега

### 4. Избыточные permissions

Если `GITHUB_TOKEN` имеет больше прав, чем нужно, атакующий при компрометации pipeline может:

- менять код репозитория
- пушить теги
- создавать релизы
- менять workflow
- воздействовать на deployment

### 5. Компрометация runner

Риски особенно возрастают на self-hosted runner:

- persistence
- lateral movement
- доступ к сети компании
- повторное использование скомпрометированной среды

### 6. Подмена артефактов

Один из важнейших supply chain рисков:

- проверили один билд
- задеплоили другой

Это ломает доверие к pipeline как к контролируемой цепочке поставки.

---

## Базовая модель secure pipeline

Нормальная цепочка выглядит так:

`Code -> Test -> Scan -> Package -> Deploy`

Смысл разделения:

- `Test` подтверждает корректность приложения
- `Scan` проверяет безопасность кода и зависимостей
- `Package` собирает артефакт
- `Deploy` выполняется только из trusted-контекста

---

## Разделение доверия

Это одна из ключевых тем DevSecOps.

### Untrusted context

- PR из fork
- внешний contributor
- код, который ещё не прошёл review

В таком контексте нельзя:

- давать секреты
- делать deploy
- давать write permissions

### Trusted context

- push в `main`
- проверенный код после review
- контролируемая ветка релиза

Здесь допустимы:

- сборка production-артефакта
- публикация
- deploy через защищённый environment

## Secure GitHub Actions pipeline

Файл workflow: `.github/workflows/ci.yml`

В pipeline реализованы такие меры:

### 1. Minimal permissions

```yaml
permissions:
  contents: read
```

Это реализация принципа least privilege. Pipeline получает только доступ на чтение содержимого репозитория.

### 2. Pinned action versions

Используются конкретные версии actions, а не плавающие теги:

- `actions/checkout@v4.3.1`
- `actions/setup-python@v6.1.0`

Это уменьшает риск supply chain подмены.

### 3. Разделение job

Workflow разбит на отдельные стадии:

- `test`
- `scan`
- `package`
- `deploy`

Это повышает прозрачность и уменьшает вероятность смешения доверенных и недоверенных шагов.

### 4. Timeouts

Для job заданы ограничения по времени:

```yaml
timeout-minutes: 10
```

Это защищает от зависших job и несанкционированно долгого выполнения.

### 5. Fail-fast

В test matrix включён `fail-fast`, чтобы не тратить ресурсы впустую после провала ключевого шага.

### 6. Checkout hardening

Используется:

```yaml
persist-credentials: false
fetch-depth: 1
```

Это уменьшает поверхность атаки и снижает риск злоупотребления токеном checkout.

### 7. Trusted deploy only

Deploy разрешён только при:

- `push`
- в ветку `main`
- через environment `production`

То есть deploy вынесен из PR-контекста.

## Что проверяет pipeline

### Test

На этапе `test`:

- устанавливаются зависимости
- запускается `pytest`

Цель: подтвердить работоспособность приложения.

### Security Scan

На этапе `scan`:

- запускается `bandit`
- запускается `pip-audit`

Цель:

- найти небезопасные конструкции в коде
- найти известные CVE и advisory в зависимостях

### Package

На этапе `package`:

- собирается Docker image

Цель:

- убедиться, что приложение реально контейнеризуется
- не допустить ситуации, когда код “тестируется”, но не собирается

### Deploy

На этапе `deploy`:

- используется environment `production`
- предусмотрен ручной approval через GitHub Environments

Цель:

- защитить production от автоматического деплоя без контроля

## Почему явный pin зависимостей важен

Даже если основная библиотека формально допускает диапазон версий транзитивной зависимости, локальная среда или Docker build могут подтянуть разные варианты. Явный pin:

- делает сборку воспроизводимой
- уменьшает нестабильность
- помогает предсказуемо проходить security checks

Это особенно важно в DevSecOps, где воспроизводимость среды напрямую влияет на доверие к pipeline.

В Dockerfile реализованы базовые меры:

## Типовые вопросы и ответы

### Почему нельзя давать pipeline write permissions?

Если pipeline будет скомпрометирован, атакующий сможет менять код, workflow, теги, релизы и артефакты. Это превращает CI в механизм компрометации supply chain.

### Чем опасны third-party actions?

Это исполняемый чужой код внутри доверенной среды. Он может получить доступ к файлам, переменным окружения и токенам. Если action будет злонамеренным или скомпрометированным, это создаёт риск утечки данных и подмены процесса сборки.

### Почему PR из fork считается риском?

Потому что код контролируется внешним автором и может выполнить вредоносные действия в CI. Если дать такому контексту секреты или write-права, последствия могут быть критичными.

### Зачем нужен protected deployment?

Чтобы отделить успешную сборку от права деплоя. Даже если build прошёл, продакшен должен оставаться под дополнительным контролем через reviewers и manual approval.
